# Обращение к команде: Обновление по модулю Парсера / Транслятора

Привет, команда! 👋

В этой ветке я подготовил базовый фундамент (MVP) для нашего транслятора (Ноды <-> C# Код). Главная цель была — создать "контракт" (структуры данных), через который наши модули смогут общаться друг с другом, а также написать основу для парсинга с использованием Roslyn.

Ниже описано, что реализовано, и как каждый из вас может это использовать.

---

## 🏗 Что было сделано

1. **Контракт данных (Промежуточное представление / IR)**
   В папке `VisualScripting.Core/Models/` лежат классы `GraphData` и `NodeData`. Это универсальный формат графа. Он ничего не знает про Unity, визуал, координаты окон или цвета кнопок. Он хранит только логику: какие ноды есть, какого они типа и как соединены.

2. **Парсер кода (Код -> Ноды)**
   В `VisualScripting.Core/Parsers/RoslynCodeParser.cs` реализован класс, который берет на вход строку C# кода, прогоняет ее через компилятор Roslyn (Microsoft.CodeAnalysis) и собирает наш объект `GraphData`.
   *Сейчас поддерживается:* Литералы (int, float, string, bool), базовые математические операции (+, -, *, /) и метод `UnityEngine.Debug.Log()`.

3. **Кодогенератор (Ноды -> Код)**
   В `VisualScripting.Core/Generators/SimpleCodeGenerator.cs` реализована обратная логика: класс принимает объект `GraphData` и рекурсивно собирает из него валидную строку C# кода.

4. **Юнит-тесты**
   Написаны тесты (`VisualScripting.Tests/ParserCodegenTests.cs`), которые проверяют полный цикл конвертации туда и обратно. Всё работает и покрыто комментариями/XML-доками на русском языке.

## 🤝 Как это использовать (по ролям)

### 👨‍💻 Разработчику №1 (Разработчик нод)
Твой главный файл для изучения — `Models/GraphData.cs`. 
Это наш с тобой "договор". 
- Когда пользователь собирает граф в редакторе, твоя система должна на выходе сформировать экземпляр `GraphData` и передать его моему генератору.
- Когда пользователь переключается из текстового кода в ноды, мой парсер вернет тебе готовый `GraphData`, и твоя задача — распарсить этот объект, создав визуальные блоки (добавив им координаты X/Y на экране, цвета и т.д.).
Давай обсудим на ближайшем созвоне: достаточно ли тебе текущих полей в `NodeData` (ID, Type, Value, InputConnections)?

### 👨‍💼 Тимлиду / Архитектору интерфейсов
Мой код выведен в отдельную независимую библиотеку `VisualScripting.Core`.
Когда ты будешь делать переключатель "Код / Ноды" в EditorWindow:
1. **Переход "Код -> Ноды":** Берешь текст из текстового поля и вызываешь `new RoslynCodeParser().Parse(text)`. Передаешь результат Разработчику №1.
2. **Переход "Ноды -> Код":** Берешь `GraphData` у Разработчика №1 и вызываешь `new SimpleCodeGenerator().GenerateCode(graph)`. Выводишь полученную строку в текстовое поле.
*Примечание:* Пока что обработка ошибок (например, если код не компилируется) в зачатке. В будущем добавим проброс ошибок синтаксиса, чтобы ты мог показывать их пользователю в UI.

### 📝 Аналитику / Тех. райтеру
У нас уже есть работающий прототип парсера! Вы можете запустить юнит-тесты командой `dotnet test` в корне проекта, чтобы убедиться, что всё работает.
Обратите внимание: мы берем только **ограниченное подмножество C#**, иначе визуальный граф превратится в кашу. Текущие ограничения MVP описаны выше. При написании документации и обучающих материалов нужно будет делать акцент на том, какие именно конструкции кода поддерживаются нашим движком.
