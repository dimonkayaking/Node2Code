### Отчёт о тестировании промежуточной версии плагина визуального скриптинга (Unity)

**Дата составления:** 19.04.2026
**Версия плагина:** 16
**Цель тестирования:** Проверка корректности генерации C# кода, удобства работы с графом и базовой функциональности редактора нод.

В ходе тестирования подтверждена работоспособность основных операций: создание переменных, условий, циклов, а также выполнение математических и логических операций. Однако выявлен ряд критических и средних дефектов, касающихся пользовательского интерфейса (UX/UI) и логики построения кода.

Ниже приведён перечень обнаруженных проблем с описанием и рекомендациями по исправлению.

---

### 1. UI/UX: Работа с вложенными структурами (Условия и Циклы)
**Описание проблемы:**
Рабочее пространство внутри нод условий и циклов реализовано неудобно:
- Маленькое окно редактирования.
- Дочерние ноды часто теряются за границами отображения (не масштабируется автоматически).
- Отсутствует возможность массового выделения нескольких нод внутри подпространства (drag selection).

**Предлагаемое решение:**
Реализовать открытие каждого подпространства (тела цикла, веток `if/else`) в **отдельном окне или на отдельной вкладке**. Это позволит работать с логикой блока как с самостоятельным графом, решив проблему навигации и масштабирования.

### 2. UI/UX: Фокус ввода в поле кода
**Описание проблемы:**
В текстовом поле для ручного написания C# кода нажатие клавиши `Tab` не вставляет символ табуляции, а вместо этого переводит фокус на следующий элемент интерфейса.

**Предлагаемое решение:**
Перехватывать событие нажатия `Tab` в поле ввода кода для вставки отступа (`\t` или 4 пробела), как это принято в стандартных редакторах кода.

### 3. Валидация данных: Отсутствие проверки типов
**Описание проблемы:**
Плагин не проверяет семантическую корректность соединений:
- Допускается сравнение значений типа `bool` и `float`.
- Допускается подключение `float` значения напрямую к порту условия (который ожидает `bool`).

**Предлагаемое решение:**
Внедрить систему проверки типов на этапе создания соединения. Если типы несовместимы, соединение должно либо запрещаться с выводом уведомления, либо подсвечиваться как ошибочное красным цветом с последующей невозможностью генерации кода.

### 4. UX: Работа с переменными
**Описание проблемы:**
Существует только одна универсальная нода для работы с переменной, что приводит к неоднозначности: в зависимости от контекста не всегда понятно, читаем мы значение переменной или записываем в неё новое.

**Предлагаемое решение:**
Разделить функционал на две отдельные ноды:
- **Get Variable** (получить значение): Только выходной порт значения.
- **Set Variable** (установить значение): Входной порт потока выполнения, входной порт значения и выходной порт потока выполнения.
Это сделает логику графа визуально прозрачнее.

### 5. Критическая ошибка: Управление потоком выполнения (Execution Flow)
**Описание проблемы:**
- Нельзя настроить порядок выполнения нод на холсте. Если на одном уровне находятся нода условия и цикла, их последовательность в сгенерированном коде не детерминирована или неочевидна для пользователя.
- У части нод есть порты `execIn` / `output`, но их функционал не реализован (соединение не влияет на порядок строк в коде).
- У большинства нод (например, арифметических) порты потока выполнения отсутствуют вообще.

**Предлагаемое решение:**
1. **Добавить порты `Exec In` и `Exec Out`** для **всех** нод, которые должны выполняться последовательно (арифметика, присвоение, вызов методов).
2. **Переименовать** порт `output` в **`execOut`** для единообразия номенклатуры.
3. При генерации кода выстраивать инструкции строго следуя пути белых (execution) соединений.

### 6. Избыточность: Логирование
**Описание проблемы:**
Ноды `Debug.Log` (специфичная для Unity) и `Console.WriteLine` (стандартная .NET) дублируют функционал вывода сообщения в консоль.

**Предлагаемое решение:**
Убрать ноду `Console.WriteLine` из основного набора, оставив только `Debug.Log`, так как плагин позиционируется как инструмент для Unity.

### 7. Сохранение: Некорректная генерация и отсутствие графа
**Описание проблемы:**
- Функция «Сохранить как» генерирует C# скрипт, который **не является валидным для Unity** (отсутствует объявление класса, наследование от `MonoBehaviour`, необходимые `using`).
- Отсутствует возможность сохранить состояние самого графа (ноды и их расположение) для последующего редактирования.

**Предлагаемое решение:**
- При генерации C# скрипта автоматически добавлять шаблон:
    ```csharp
    using UnityEngine;
    public class [ClassName] : MonoBehaviour
    {
        // Сгенерированный код
    }
    ```
- Реализовать отдельную команду «Сохранить граф» (в собственном формате `.asset` или `.json`).

### 8. Скрытый функционал: Вход execIn у Debug.Log
**Описание проблемы:**
У ноды `Debug.Log` существует порт входа потока выполнения (`execIn`), но он **скрыт визуально** на ноде. Обнаружить его можно только при попытке протянуть соединение от выхода другой ноды и выбрав `Debug.Log: execIn` в выпадающем меню.

**Предлагаемое решение:**
Скорректировать шаблон отображения ноды `Debug.Log`, чтобы порт `execIn` был виден явно.

### 9. Баг интерфейса: Отображение нод Unity
**Описание проблемы:**
При создании новой ноды через стандартное контекстное меню (или меню создания) в списке категорий отсутствует блок **Unity** (ноды для работы с векторами, `Transform` и т.д.). Однако при создании ноды через вытягивание соединения из порта другой ноды, этот блок появляется и позволяет создать, например, `Vector3`.

**Предлагаемое решение:**
Унифицировать источник данных для обоих меню. Добавить недостающую категорию **Unity** в общее меню создания нод на холсте.

### Заключение
**Общая оценка версии:** Условно-рабочая. Основной каркас визуального скриптинга функционирует.
**Приоритеты доработок:**
1. **Высокий:** Реализация Exec входов/выходов у всех нод (п.5), Валидация типов (п.3), Исправление генерации C# скрипта (п.7).
2. **Средний:** Рефакторинг работы с переменными (п.4), Устранение багов с отображением нод (п.9) и скрытыми портами (п.8).
3. **Низкий:** Улучшения UX (п.1, п.2), Удаление лишних нод (п.6).

После внесения указанных правок рекомендуется проведение повторного регрессионного тестирования.
